The AI Agent Control Plane
Agentic Security Overview

Securing AI Agents at Enterprise Scale

AI agents are becoming the new norm — answering questions, taking actions, and reaching across ERP, CRM, code, and data. Legacy IAM was built for humans and deterministic systems; it has no answer for autonomous, non-deterministic agents acting on a user's behalf. The AI Agent Control Plane closes that gap with four pillars: identity, traceability, runtime protection, and observability.

WHAT

A unified control plane that gives every AI agent a unique, verifiable identity, a full delegation chain, guardrails on every request, and end-to-end observability — with zero application code changes.

WHY

Autonomous agents create new non-deterministic risk. Shared identities, invisible delegation, unfiltered prompts, and black-box decisions turn a single prompt injection into a data-exfiltration path that traditional logs never catch.

WHO

Platform, security, and SRE teams operating AI agents against regulated data. Architects who need to prove who did what, for whom, under what policy — on every call, for every replica.

Where agentic AI breaks down without the right foundations

Four failure modes that legacy controls cannot see, let alone stop.

Non-deterministic behavior

Same name, different behavior. Two replicas of the same agent can reach different decisions while sharing one identity.

  • Shared identity masks which replica acted.
  • Impossible to audit or hold the right instance accountable.

Invisible delegation

Chains of accountability collapse. Should the agent act as the user, or as itself? The wrong choice risks impersonation or over-privileged access.

  • No audit trail across agent handoffs.
  • Impossible to trace who did what, when.

No real-time enforcement

Guardrails only exist in the developer's mind. Prompt injection and unsafe tool use reach production unfiltered.

  • Data leakage via outputs is unchecked.
  • No runtime policy — no control over what agents can't do.

No central visibility

Flying blind at enterprise scale. Policy violations go undetected. Agent workflows cross multiple systems without unified tracing.

  • Limited insight into agent decisions, tool usage, and actions.
  • No end-to-end view from human intent to final side effect.

Real-world risk: this is already happening

Without proper guardrails, AI agents with infrastructure access are already causing production incidents.

"Claude Code wiped our production database with a Terraform command. It took down the course platform and 2.5 years of submissions: homework, projects, and leaderboards. Automated snapshots were gone too."
— Alexey Grigorev · DataTalksClub

What was missing:

  • Unrestricted access — the agent could run any Terraform command against production, including destroy.
  • No human gate — destructive changes executed without approval or review.
  • No environment isolation — the agent operated directly against production; no sandbox, staging, or workspace separation.

Guardrails aren't optional. Every risk above needs to be addressed by controls enforced at the platform layer — not left to application code.

Chapter Two

The AI Agent Control Plane

Identity proves who acts. Traceability proves for whom. Runtime policy constrains what the agent may do. Observability proves it after the fact.

Inside the control plane — how it all fits together

Three layers: identity & traceability, runtime protection, and observability — with data flowing from user to audit in one continuous chain.

A tale of four identities

Every time an AI agent acts on a user's behalf, four identities are in play. Each one answers a different trust question.

User identity

also called subject_token
Proves
Who is asking.
Issued by
Corporate login — IBM Verify, Entra ID.
name:  "Alice Johnson"
email: "alice@bank.com"
role:  "loan_officer"
iss:   "verify.ibm.com"

Platform identity

K8s SA · Nomad workload id
Proves
Where the agent is running.
Issued by
Kubernetes / Nomad — automatic.
pod:  "loan-agent-7a2f"
ns:   "banking-prod"
sa:   "loan-agent-sa"
node: "worker-03"

Agent identity

also called actor_token
Proves
Which exact agent replica — unique per instance.
Issued by
Vault — the identity broker.
agent_id: "7a2f-loan-bot"
entity:   "e8f2-a3b1"
ttl:      "1h"
iss:      "vault.bank.internal"

On-Behalf-Of token

delegated · user × agent
Proves
Agent-7a2f acting on behalf of Alice — both at once.
Issued by
IDP — via token exchange (RFC 8693).
sub:   "alice@bank.com"
act:   "7a2f-loan-bot"
scope: "loans.read"
aud:   "loan-api.bank.internal"

Banking example. Alice asks the loan-status agent about customer #12345. The Loan API sees that Alice requested it, agent 7a2f carried it out, and access is scoped to loans.read only — nothing else.

Human meets AI agent — the identity handshake

The user proves who they are. The agent proves which instance it is. Token exchange turns both into a single scoped credential.

In practice. Alice logs into the banking app. Separately, the loan-status agent receives its identity from Vault. When the agent calls the Loan API on Alice's behalf, both credentials combine into one — so the API knows who asked and which agent carried it out.

Chapter Three
Pillar 1 · Identity

It all starts with unique identity

Every agent replica gets its own signed, verifiable identity — automatically, with zero application code changes.

Without unique identity, trust is impossible

Without unique identity

Both replicas log as "LoanAgent"

  • Replica A approves a loan. Replica B declines it. Logs can't tell which was which.
  • Impossible to audit which replica acted.
  • Compliance has no actor-level evidence.
  • Downstream trust decisions start from a blind spot.
With the control plane

Two replicas → two signed identities

  • Platform bootstrap — each agent inherits platform-native identity automatically.
  • Identity broker mints an OIDC actor_token per replica — unique, short-lived, no static secrets.
  • Every action is attributable: downstream systems verify the token and tie each decision to the exact replica.

Automatic agent identity injection — how it works

Four steps. No secrets to manage. No code changes. Every agent replica uniquely identified.

In this repo

Where identity is implemented. Click to expand.

deploy-k8s/ Vault Agent injector annotations (K8s path)

Pods are annotated to request Vault Agent sidecar injection. The sidecar authenticates with the pod's ServiceAccount JWT and writes the actor_token to /vault/path/actor-token:

  • vault.hashicorp.com/role: "agent-role"
  • vault.hashicorp.com/auth-type: "jwt"
  • vault.hashicorp.com/agent-inject-secret-actor-token: "identity/oidc/token/agent-role"
ai-agent/ FastAPI runtime reads the injected token

The agent runtime reads the rotating actor_token from disk at request time and presents it as proof of which replica is calling in every downstream OBO exchange. No secrets are baked into the image; Vault Agent refreshes the file in place.

infra/ Vault JWT auth role + OIDC issuer config

Terraform provisions the Vault JWT auth backend that trusts the K8s or Nomad issuer, the OIDC key used to sign actor_tokens, and the role that binds the two. The IDP is configured to trust Vault as an OIDC issuer for agent identities — so token exchange at the IDP can verify the actor_token natively.

Chapter Four
Pillar 2 · Traceability

Know who did what — and for whom

Every agent action is cryptographically tied to the human who requested it. Delegation preserved across every hop.

Without a delegation chain, identity gets lost at every hop

Without OBO

User → Agent → Action

  • Impersonation — agent acts as the user, no scope constraint.
  • Privilege escalation — over-privileged access with no scope limit.
  • No audit evidence — compliance can't prove who did what, when.
With OBO · RFC 8693

User delegates → Agent acts → Action traceable

  • User identity preserved — subject stays across every hop.
  • Least privilege — scoped, short-lived actor tokens only.
  • Tamper-proof log — who, to whom, when, under what constraints.

Traceability — how it works

Four stages turn subject + actor into a single signed on-behalf-of credential.

OBO token anatomy

{ "act": { "bu": "hr", "department": "ops", "service_group": "employee-mgmt" }, "actor": { "agentId": "emp-agent-a12" }, "sub": "alice", "scope": "agent.invoke" }

Vault's role. Vault issues the short-lived actor_token to each agent workload based on its platform identity. The IDP is configured to trust Vault as an OIDC issuer for agent identities. Revoke at Vault → all future token exchanges fail.

In this repo

token-exchange/ FastAPI OBO broker (RFC 8693)
Receives subject_token + actor_token and exchanges them with IBM Verify for a delegated OBO access token. Returns it to the agent for downstream API calls.
web-app/auth/oauth.py captures subject_token at login
Streamlit UI authenticates the user via IBM Verify OAuth, captures the issued subject_token in the session, and passes it to the ai-agent on every request.
ai-agent/ combines subject + actor → OBO call
The agent reads the Vault-injected actor_token, takes the caller's subject_token, and calls token-exchange to obtain the delegated OBO credential. Every downstream call carries both identities.
Chapter Five
Pillar 3 · Runtime Protection

Stop threats before they reach your agent

Every request and response inspected against policy — enforced at the platform layer, on every invocation, with zero agent code changes.

Without runtime protection, every threat passes straight through

Unguarded agent

Threats in · harmful outputs out

  • Prompt injection, jailbreak attempts, harmful requests, unsafe tool calls — all reach the agent unfiltered.
  • PII exposure, prompt leakage, harmful content — all leave the system unchecked.
  • No interception anywhere.
Platform-enforced guardrail

Intercept · evaluate · allow, mask, or block

  • Pre-check blocks injection, jailbreak, harmful intent before the agent sees it.
  • Post-check masks PII, blocks data leakage, enforces policy violations before the response leaves.
  • Pluggable policy engine — rule-based (OPA) and ML-based (watsonx.governance).
  • Zero code changes to the agent. Policies updated centrally.

The guardrail — how it works

Every API call validated against policy. Nothing executes outside boundaries.

Inside the AI guardrail

Three components wired together — intercept, evaluate, store.

A · Traffic Interceptor

Consul Envoy sidecar

Inspects every request and response. Pre-request: detects prompt injection, blocks unsafe or non-compliant prompts. Post-request: filters responses, masks PII, blocks harmful content.

Consul
B · Policy Engine

Pluggable — OPA and/or wx.gov

OPA for rule-based, deterministic, auditable policy. watsonx.governance for AI-powered, multi-category probabilistic scoring. Run one or combine.

Consul
C · Policy & Credential Store

HashiCorp Vault

Single source of truth for policy rules — versioned and audited. Issues scoped credentials on demand; expires in minutes; matches what the user consented to.

Vault
